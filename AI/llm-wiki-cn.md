# LLM Wiki

一种使用大语言模型构建个人知识库的模式。

这是一个想法文档，旨在被复制粘贴到你自己的 LLM 代理（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）中使用。它的目标是传达高层级的概念，而你的代理将与你协作，构建出具体的实现细节。

## 核心理念

大多数人对 LLM 和文档的体验类似于 RAG：你上传一批文件，LLM 在查询时检索相关片段，然后生成答案。这种方式是可行的，但 LLM 在每个问题上都要从头重新发现知识。没有任何积累。问一个需要综合五个文档的微妙问题，LLM 就必须每次都找到并拼凑出相关的片段。没有任何东西被构建起来。NotebookLM、ChatGPT 文件上传以及大多数 RAG 系统都是以这种方式工作的。

这里的理念有所不同。LLM 并不只是在查询时从原始文档中检索，而是**增量式地构建并维护一个持久化的 wiki** —— 一个结构化的、相互链接的 Markdown 文件集合，位于你和原始来源之间。当你添加一个新来源时，LLM 不仅仅是对其进行索引以便日后检索。它会读取该来源，提取关键信息，并将其整合到现有的 wiki 中 —— 更新实体页面、修订主题摘要、记录新数据与旧主张相矛盾的地方，从而加强或挑战不断演进的综合理解。知识被编译一次，然后*保持更新*，而不是在每次查询时重新推导。

这是关键区别：**wiki 是一个持久的、不断累积的产物。** 交叉引用已经存在。矛盾之处已经被标记。综合理解已经反映出你所阅读过的所有内容。随着你添加的每一个来源和提出的每一个问题，wiki 会变得越来越丰富。

你从不（或很少）自己编写 wiki —— LLM 负责编写和维护所有内容。你负责 sourcing（寻找/收集资料）、探索和提出正确的问题。LLM 完成所有繁重的工作 —— 总结、交叉引用、归档和簿记，这些工作让知识库随着时间的推移真正变得有用。在实践中，我会让 LLM 代理在一侧打开，Obsidian 在另一侧打开。LLM 根据我们的对话进行编辑，我则实时浏览结果 —— 点击链接、查看关系图、阅读更新后的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这种方式可以应用于许多不同的场景。几个例子：

- **个人**：跟踪你自己的目标、健康、心理、自我提升 —— 归档日记条目、文章、播客笔记，并随着时间的推移构建一个结构化的自我画像。
- **研究**：在数周或数月内深入某个主题 —— 阅读论文、文章、报告，并逐步构建一个包含不断演进的论点的综合性 wiki。
- **阅读一本书**：随着阅读进度归档每一章，为人物、主题、情节线索及其联系构建页面。到最后，你会拥有一个丰富的配套 wiki。想想像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki —— 由一群志愿者多年构建的成千上万个相互链接的页面，涵盖人物、地点、事件、语言。你可以在阅读时亲自构建这样的东西，而 LLM 负责所有的交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部 wiki，输入来源包括 Slack 对话、会议记录、项目文档、客户通话。可能还需要人工参与审核更新。因为 LLM 完成了团队中没人想做的维护工作，wiki 能够保持最新状态。
- **竞争分析、尽职调查、旅行规划、课程笔记、兴趣深度研究** —— 任何你需要随时间积累知识并希望得到组织整理而非零散分布的事情。

## 架构

分为三层：

**原始来源** —— 你精心挑选的源文档集合。文章、论文、图片、数据文件。这些是不可变的 —— LLM 从中读取但从不修改它们。这是你的真实来源。

**wiki** —— 一个由 LLM 生成的 Markdown 文件目录。包含摘要、实体页面、概念页面、对比、概览、综合总结。LLM 完全拥有这一层。它创建页面，在新来源到来时更新它们，维护交叉引用，并保持一切一致性。你阅读它；LLM 编写它。

**schema（模式）** —— 一个文档（例如，针对 Claude Code 的 CLAUDE.md 或针对 Codex 的 AGENTS.md），它告诉 LLM wiki 是如何组织的，约定是什么，以及在摄取来源、回答问题或维护 wiki 时应遵循的工作流程。这是关键的配置文件 —— 它使 LLM 成为一个有纪律的 wiki 维护者，而不是一个通用的聊天机器人。你和 LLM 共同演进这个文件，因为你会发现什么在你的领域中有效。

## 操作

**Ingest（摄取）**。你将一个新的来源放入原始来源集合，并告诉 LLM 处理它。一个示例流程：LLM 读取来源，与你讨论关键要点，在 wiki 中编写一个摘要页面，更新索引，更新 wiki 中相关的实体和概念页面，并在日志中追加一个条目。一个源文件可能会触及 10-15 个 wiki 页面。我个人更喜欢逐个摄取来源并保持参与 —— 我会阅读摘要、检查更新，并指导 LLM 应该强调什么。但你也可以一次性批量摄取多个来源，减少监督。这取决于你，你需要开发适合你风格的工作流程，并将其记录在 schema 中，供后续会话使用。

**Query（查询）**。你向 wiki 提出问题。LLM 搜索相关页面，阅读它们，并综合出一个带有引用的答案。根据问题的不同，答案可以采用不同的形式 —— 一个 Markdown 页面、一个对比表格、一个幻灯片（Marp）、一个图表（matplotlib）、一个画布。重要的见解是：**好的答案可以作为一个新页面归档回 wiki 中。** 你要求做的对比、分析、你发现的联系 —— 这些都是有价值的，不应该消失在聊天记录中。这样，你的探索就会像被摄取的来源一样，在知识库中不断累积。

**Lint（检查/清理）**。定期要求 LLM 对 wiki 进行健康检查。寻找：页面之间的矛盾、已被新来源取代的过时主张、没有入链的孤立页面、被提及但缺少独立页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 善于建议要研究的新问题和要寻找的新来源。这有助于 wiki 在增长过程中保持健康。

## 索引与日志

两个特殊文件帮助 LLM（以及你）在 wiki 增长时进行导航。它们有不同的用途：

**index.md** 是面向内容的。它是 wiki 中所有内容的目录 —— 每个页面都列出链接、一行摘要，以及可选的元数据（如日期或来源数量）。按类别（实体、概念、来源等）组织。LLM 在每次摄取时都会更新它。当回答查询时，LLM 首先读取索引以找到相关页面，然后深入阅读它们。这在中等规模（约 100 个来源，数百个页面）下效果出奇地好，并且避免了基于嵌入向量的 RAG 基础设施的需求。

**log.md** 是按时间顺序排列的。它是一个只追加（append-only）的记录，记录发生的事情和发生时间 —— 包括摄取、查询、检查等操作。一个有用的技巧：如果每个条目都以一致的前缀开头（例如，`## [2026-04-02] ingest | Article Title`），那么日志就可以用简单的 Unix 工具解析 —— `grep "^## \[" log.md | tail -5` 会给你最后 5 条条目。日志提供了 wiki 演化的时间线，并帮助 LLM 理解最近完成了哪些工作。

## 可选：CLI 工具

在某个时候，你可能希望构建一些小工具，帮助 LLM 更有效地操作 wiki。一个搜索 wiki 页面的搜索引擎是最明显的 —— 在规模较小时，索引文件就足够了，但随着 wiki 的增长，你会需要合适的搜索。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是一个针对 Markdown 文件的本地搜索引擎，具有混合 BM25/向量搜索和 LLM 重排序功能，全部在本地运行。它既有 CLI（LLM 可以通过 shell 调用它），也有 MCP 服务器（LLM 可以将其作为原生工具使用）。你也可以自己构建一些更简单的东西 —— 当需要时，LLM 可以帮助你通过 vibe-coding 快速编写一个简单的搜索脚本。

## 技巧与窍门

- **Obsidian Web Clipper** 是一个浏览器扩展，可以将网页文章转换为 Markdown。对于快速将来源获取到你的原始来源集合中非常有用。
- **本地下载图片。** 在 Obsidian 设置 → 文件和链接中，将“附件文件夹路径”设置为一个固定目录（例如 `raw/assets/`）。然后在设置 → 快捷键中，搜索“Download”以找到“下载当前文件的附件”，并为其绑定一个快捷键（例如 Ctrl+Shift+D）。剪藏一篇文章后，按下该快捷键，所有图片都会被下载到本地磁盘。这是可选的，但很有用 —— 它让 LLM 可以直接查看和引用图片，而不是依赖可能失效的 URL。请注意，LLM 不能原生地一次性阅读带有内联图片的 Markdown —— 解决方法是让 LLM 先阅读文本，然后单独查看部分或全部引用的图片以获取额外的上下文。这有点笨拙，但效果还不错。
- **Obsidian 的关系图视图** 是查看你的 wiki 结构的最佳方式 —— 什么连接到什么，哪些页面是集线器，哪些是孤岛。
- **Marp** 是一种基于 Markdown 的幻灯片格式。Obsidian 有一个插件支持它。用于直接从 wiki 内容生成演示文稿非常有用。
- **Dataview** 是一个 Obsidian 插件，可以对页面 frontmatter 运行查询。如果你的 LLM 向 wiki 页面添加了 YAML frontmatter（标签、日期、来源数量），Dataview 可以生成动态表格和列表。
- wiki 就是一个 Markdown 文件的 git 仓库。你可以免费获得版本历史、分支和协作功能。

## 为什么这种方法有效

维护知识库的繁琐部分不是阅读或思考 —— 而是簿记。更新交叉引用、保持摘要最新、记录新数据与旧主张的矛盾之处、在几十个页面之间保持一致性。人类放弃 wiki 是因为维护负担的增长速度超过了价值增长的速度。LLM 不会感到无聊，不会忘记更新交叉引用，并且可以一次性处理 15 个文件。wiki 能够保持维护状态，因为维护成本几乎为零。

人类的工作是策划来源、指导分析、提出好的问题，以及思考这一切意味着什么。LLM 的工作是其他所有事情。

这个想法在精神上与 Vannevar Bush 的 Memex（1945）有关 —— 一个个人化的、精心策划的知识库，文档之间具有关联路径。Bush 的愿景比后来网络变成的样子更接近于这个想法：私人的、主动管理的，文档之间的联系与文档本身同样有价值。他无法解决的部分是：谁来负责维护。LLM 处理了这个问题。

## 注意

本文档特意是抽象的。它描述的是理念，而不是具体的实现。具体的目录结构、schema 约定、页面格式、工具 —— 所有这些都取决于你的领域、你的偏好以及你选择的 LLM。上面提到的所有内容都是可选的、模块化的 —— 选取有用的，忽略无用的。例如：你的来源可能只有文本，所以你根本不需要图像处理。你的 wiki 可能规模很小，索引文件就足够了，不需要搜索引擎。你可能不关心幻灯片，只想要 Markdown 页面。你可能想要一套完全不同的输出格式。使用本文档的正确方式是将其分享给你的 LLM 代理，并一起工作，实例化一个适合你需求的版本。本文档的唯一工作就是传达这个模式。你的 LLM 可以处理剩下的事情。
